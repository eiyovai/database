## 1. 项目概述

本系统为**校园一体化访客管理数据库系统**，整合了校园开放预约、访客权限管控、校园活动报名、客流容量控制、安全审计追溯等核心功能。系统面向**校外访客**、**校园管理员**、**安保人员**、**志愿工作人员**四类角色，实现访客入校全流程数字化、规范化、智能化管理。

系统核心业务流程为：

```mermaid

    B -->|正常| C[提交入校预约]
    C --> C1[INSERT Reservations<br/>Status='pending']
    C1 --> C2[触发器: INSERT ReservationStatusLog]
    C2 --> D[管理员审核]

    D1 -->|通过| D2[UPDATE Reservations<br/>Status='approved']

    D2 --> D4[触发器: INSERT StatusLog]
    D2 --> E[访客按时到校]
```

**业务流程对应数据库表**：

| 流程节点   | 涉及数据库表                                           | 关键操作                                                            |
| ---------- | ------------------------------------------------------ | ------------------------------------------------------------------- |
| 黑名单检查 | `Blacklist`                                          | `SELECT WHERE UserId=? AND IsActive=1 AND ExpiresAt>GETUTCDATE()` |
| 预约提交   | `Reservations` + `ReservationStatusLog`            | INSERT + 触发器自动写日志                                           |
| 预约审核   | `Reservations` + `ReservationStatusLog`            | UPDATE Status + 触发器                                              |
| 入校核验   | `Reservations` + `EntryExitRecords` + `Gates`    | UPDATE + INSERT（同一事务）                                         |
| 离校登记   | `EntryExitRecords` + `Reservations`                | UPDATE ExitTime + UPDATE Status                                     |
| 爽约检测   | `Reservations` + `ViolationRecords`                | 查询过期approved → INSERT违规                                      |
| 超时检测   | `EntryExitRecords` + `OpenRules` + `CampusAreas` | 按区域关闭时间判断滞留                                              |
| 自动拉黑   | `Blacklist`                                          | `INSERT WHERE 累计违规≥3`                                        |
| 举报处理   | `Reports` → `ViolationRecords` → `Blacklist`   | 审核通过→违规记录→拉黑链                                          |

## 2. 需求分析

### 2.1 用户角色分析

| 角色                          | 职责               | 核心需求                                                                            |
| ----------------------------- | ------------------ | ----------------------------------------------------------------------------------- |
| **校外访客** (visitor)  | 预约入校、报名活动 | 提交预约、查看审核状态、浏览活动、取消预约                                          |
| **管理员** (admin)      | 全局管理与配置     | 审核预约、管理开放规则、管理区域/校门、活动管理、黑名单管理、排班管理、查看数据看板 |

### 2.3 非功能需求

- **安全性**：密码 BCrypt 哈希存储，JWT 令牌认证，基于角色的访问控制（RBAC）
- **数据完整性**：外键约束、CHECK 约束、唯一约束保证数据一致性
- **可追溯性**：预约状态变更触发器自动记录日志，审计日志记录所有后台操作
- **可扩展性**：分层架构（Controller → Service → Repository/EF Core），便于后续扩展

### 3.2 架构选型理由

采用**前后端分离**架构的核心考量：

1. **开发解耦**：前端和后端可以独立开发和调试，互不阻塞
2. **多端复用**：后端 API 可同时服务 Web 端、移动端等不同前端
3. **职责清晰**：后端专注业务逻辑和数据访问，前端专注用户交互和界面展示
4. **符合行业趋势**：前后端分离是当前 Web 开发的主流模式

## 5. 数据库设计

> **数据库名称**：`CampusVisitorDB`
> **数据库引擎**：SQL Server 2019
> **总表数量**：16 张
> **设计原则**：第三范式（3NF），合理的冗余以优化查询性能

### 5.1 概念结构设计（ER 图）

```mermaid
erDiagram
    Users ||--o{ Visitors : "1:1 扩展"
    Users ||--o{ Reservations : "创建预约"
    Users ||--o{ EntryExitRecords : "出入记录"
    Users ||--o{ AuditLogs : "产生日志"
    Users ||--o{ StaffSchedules : "排班安排"
    Users ||--o{ ViolationRecords : "违规记录"
    Users ||--o{ Blacklist : "被拉黑"
    Users ||--o{ ActivityRegistrations : "报名活动"
    Users ||--o{ Reports : "提交举报"

    Reservations ||--o{ ReservationStatusLog : "状态追溯"
    Reservations ||--o{ EntryExitRecords : "出入关联"

    CampusAreas ||--o{ AreaPermissions : "区域权限"
    CampusAreas ||--o{ OpenRules : "区域开放规则"

    Activities ||--o{ ActivityRegistrations : "活动报名"

    Gates ||--o{ EntryExitRecords : "入校校门"
    Gates ||--o{ EntryExitRecords : "离校校门"

    Users ||--o{ Reservations : "审核预约"
    Users ||--o{ Reports : "审核举报"
  
    OpenRules ||--o| CampusAreas : "绑定区域"
    ViolationRecords }o--|| Reservations : "关联预约"
    ViolationRecords }o--|| EntryExitRecords : "关联出入记录"
```

**ER 图说明**：

- `Users` 是系统的**中心实体**，统一管理所有角色（访客、管理员、安保、工作人员），通过 `Role` 字段区分
- `Visitors` 是 `Users` 的 **1:1 扩展表**，存放访客特有信息（身份证号、访客类型等）
- `Reservations` 是**核心业务表**，承载预约申请、审核、入校、离校全流程
- `ReservationStatusLog` 是**审计关联表**，通过触发器自动记录每一次状态变更
- `CampusAreas` 和 `AreaPermissions` 构成**多对多关系**，实现不同访客类型对不同区域的差异化访问权限
- `OpenRules` 通过 `AreaId` 外键关联到 `CampusAreas`，实现**区域级差异化开放规则配置**，NULL 表示全校通用规则
- `EntryExitRecords` 同时关联 `Reservations`、`Users`、`Gates`（入口/出口），形成完整的出入轨迹
- `ViolationRecords` 通过 `SourceId` 关联至预约表或出入记录表，实现**违规溯源**

### 5.3 物理结构设计

#### 5.3.1 索引设计

为提高查询性能，在以下高频查询字段上建立了非聚集索引：

```sql
-- 预约表：按用户、状态、日期、手机号查询
CREATE NONCLUSTERED INDEX [IX_Reservations_UserId]        ON Reservations(UserId)
CREATE NONCLUSTERED INDEX [IX_Reservations_Status]        ON Reservations(Status)
CREATE NONCLUSTERED INDEX [IX_Reservations_VisitDate]     ON Reservations(VisitDate)
CREATE NONCLUSTERED INDEX [IX_Reservations_VisitorPhone]  ON Reservations(VisitorPhone)

-- 状态日志表：按预约ID追溯
CREATE NONCLUSTERED INDEX [IX_RSL_ReservationId]          ON ReservationStatusLog(ReservationId)

-- 出入校记录：按用户、入校时间查询
CREATE NONCLUSTERED INDEX [IX_EER_UserId]                 ON EntryExitRecords(UserId)
CREATE NONCLUSTERED INDEX [IX_EER_EntryTime]              ON EntryExitRecords(EntryTime)

-- 活动报名：按活动ID统计
CREATE NONCLUSTERED INDEX [IX_AR_ActivityId]              ON ActivityRegistrations(ActivityId)

-- 违规记录：按用户查询违规历史
CREATE NONCLUSTERED INDEX [IX_VR_UserId]                  ON ViolationRecords(UserId)

-- 审计日志：复合查询优化
CREATE NONCLUSTERED INDEX [IX_AL_OperatorId]              ON AuditLogs(OperatorId)
CREATE NONCLUSTERED INDEX [IX_AL_CreatedAt]               ON AuditLogs(CreatedAt)
CREATE NONCLUSTERED INDEX [IX_AL_ActionType]              ON AuditLogs(ActionType)

-- 举报：按状态筛选
CREATE NONCLUSTERED INDEX [IX_Rep_Status]                 ON Reports(Status)

-- 排班：按日期查询
CREATE NONCLUSTERED INDEX [IX_SS_WorkDate]                ON StaffSchedules(WorkDate)

-- 开放规则：按区域+日期+状态复合查询
CREATE NONCLUSTERED INDEX [IX_OR_AreaId]                  ON OpenRules(AreaId)
CREATE NONCLUSTERED INDEX [IX_OR_IsActive]                ON OpenRules(IsActive)
```

**索引设计理由**：

- `Reservations` 是核心查询表，按 `Status`（审核列表）、`VisitDate`（当日预约）、`VisitorPhone`（安保核验）是最高频的查询条件
- `AuditLogs` 表数据量会随时间线性增长，必须对 `CreatedAt` 和 `ActionType` 建立索引以支持按时间和类型筛选
- `EntryExitRecords` 查询"当前在校访客"需要 `EntryTime IS NOT NULL AND ExitTime IS NULL`，对 `EntryTime` 建立索引可加速该过滤

#### 5.3.2 触发器设计

**核心触发器**：`trg_Reservations_StatusChange` — 预约状态变更自动记录日志

```sql
CREATE TRIGGER [dbo].[trg_Reservations_StatusChange]
ON [dbo].[Reservations]
AFTER UPDATE
AS
BEGIN
    SET NOCOUNT ON
    -- 仅当 Status 字段被更新时触发
    IF UPDATE([Status])
    BEGIN
        INSERT INTO [dbo].[ReservationStatusLog]
            ([ReservationId], [FromStatus], [ToStatus], [OperatorId], [Remark])
        SELECT
            i.[Id],                          -- 预约ID
            d.[Status],                      -- 变更前状态（来自 DELETED 伪表）
            i.[Status],                      -- 变更后状态（来自 INSERTED 伪表）
            i.[ReviewerId],                  -- 操作人
            CASE i.[Status]
                WHEN 'approved'   THEN N'审核通过'
                WHEN 'rejected'   THEN N'审核拒绝: ' + ISNULL(i.[ReviewRemark], N'')
                WHEN 'cancelled'  THEN N'用户取消预约'
                WHEN 'checked_in' THEN N'已入校核验'
                WHEN 'checked_out' THEN N'已离校登记'
                WHEN 'no_show'    THEN N'系统标记爽约'
                ELSE N'状态变更'
            END
        FROM INSERTED i
        INNER JOIN DELETED d ON i.[Id] = d.[Id]
    END
END
GO
```

**触发器设计说明**：

- 利用 SQL Server 的 `INSERTED` 和 `DELETED` 伪表对比获取状态变更前后的值
- 使用 `IF UPDATE(Status)` 条件判断，仅在 Status 字段变更时才执行插入，避免不必要的日志写入
- 自动根据目标状态生成中文备注，减少应用层代码负担
- 这意味着**任何途径**（无论是前端 API 调用还是直接 SQL 操作）修改预约状态，都会自动记录日志，保证审计完整性

#### 5.3.2.1 为什么系统只有一个触发器？

系统中除了日志记录，还有**违规检测、自动拉黑、活动过期关闭**等自动化逻辑。这些**不是用触发器实现**，而是用 **C# 代码**，原因如下：

| 自动化逻辑         | 实现方式                          | 为何不用触发器                                                                                                                              |
| ------------------ | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 预约状态变更日志   | SQL 触发器 ✅                     | 简单的 INSERT 操作，适合触发器。任何路径修改状态都能被捕获                                                                                  |
| 爽约/超时/越权检测 | C#`AutoDetectViolationsAsync()` | 需要**跨多表关联查询**（`Reservations` + `OpenRules` + `CampusAreas` + `EntryExitRecords`），SQL 触发器中写复杂 JOIN 难以维护 |
| 自动拉黑           | C#`AutoBlacklistAsync()`        | 需要在违规检测完成后**先 SaveChanges 再统计**，触发器无法控制事务提交顺序                                                             |
| 活动过期关闭       | C#`AutoCloseExpiredAsync()`     | 仅在查询时触发（懒检查），不需要实时轮询。触发器无法响应"时间到了"这个事件                                                                  |

**关键原则**：SQL 触发器适合**单表、简单、必须无条件执行**的审计操作。涉及多表关联、业务规则判断、需要在特定时机（页面访问时）触发的逻辑，统一用 C# Service 层实现，代码更清晰、更易调试。

| 对比维度 | SQL 触发器                         | C# 业务逻辑         |
| -------- | ---------------------------------- | ------------------- |
| 适用场景 | 审计日志、数据级联                 | 业务规则、多表查询  |
| 可调试性 | 难以调试（黑盒）                   | IDE 断点调试        |
| 可测试性 | 需构造 SQL 场景                    | 单元测试 + 集成测试 |
| 维护性   | DBA 维护                           | 全栈开发者维护      |
| 触发时机 | 数据变更时（INSERT/UPDATE/DELETE） | 页面访问/API 调用时 |

#### 5.3.3 自定义函数设计

**预约编号生成函数**：

```sql
CREATE FUNCTION [dbo].[GenerateReservationNo]()
RETURNS VARCHAR(30)
AS
BEGIN
    DECLARE @dateStr VARCHAR(10) = FORMAT(GETDATE(), 'yyyyMMdd')
    DECLARE @seq INT

    -- 查询当日最大序号并 +1
    SELECT @seq = ISNULL(MAX(CAST(RIGHT([ReservationNo], 4) AS INT)), 0) + 1
    FROM [dbo].[Reservations]
    WHERE [ReservationNo] LIKE 'R' + @dateStr + '%'

    -- 返回格式: R + 日期 + 4位序号，如 R202606260001
    RETURN 'R' + @dateStr + RIGHT('0000' + CAST(@seq AS VARCHAR), 4)
END
GO
```

**设计要点**：

- 预约编号格式：`R` + 8位日期 + 4位流水号（如 `R202606260001`）
- 按日期独立编号，每天从 `0001` 开始
- 使用 `ISNULL(MAX(...), 0) + 1` 确保即使当日无预约也能正确从 1 开始
- 配合 `ReservationNo` 字段的 UNIQUE 约束，保证编号全局唯一

#### 5.3.4 约束汇总

| 约束类型                 | 数量 | 示例                                                                           |
| ------------------------ | ---- | ------------------------------------------------------------------------------ |
| **PRIMARY KEY**    | 16   | 每表一个自增主键                                                               |
| **FOREIGN KEY**    | 26+  | `Reservations.UserId → Users.Id`、`OpenRules.AreaId → CampusAreas.Id` 等 |
| **UNIQUE**         | 8    | `Users.Phone`、`Reservations.ReservationNo`、`Blacklist.UserId` 等       |
| **CHECK**          | 15+  | `Users.Role IN (...)`、`Reservations.Status IN (...)` 等                   |
| **DEFAULT**        | 20+  | `Users.Role DEFAULT 'visitor'`、`Reservations.Status DEFAULT 'pending'` 等 |
| **CASCADE DELETE** | 1    | `AreaPermissions.AreaId → CampusAreas.Id`（删除区域时级联删除权限）         |

**关键约束设计说明**：

1. **CHECK 约束的枚举值控制**：所有状态字段（`Role`、`Status`、`VisitorType`、`TimeSlot` 等）都使用 CHECK 约束限制取值，确保数据层面不会出现非法状态值，比应用层校验更可靠
2. **ON DELETE NO ACTION**：大部分外键使用 `NO ACTION`（默认行为），防止误删导致级联数据丢失，例如删除用户时不允许直接删除（因为有关联的预约记录），需要业务层先处理
3. **ON DELETE CASCADE**：仅在 `AreaPermissions` 上使用，因为权限是区域的附属属性，删除区域时权限也失去意义

### 6.3 核心业务逻辑详解

#### 6.3.1 预约创建流程

```csharp
// ReservationService.CreateAsync()
public async Task CreateAsync(int userId, CreateReservationRequest request)
{
    // 1. 查询当日最后一条预约编号，生成新的流水号
    var last = await _db.Reservations
        .OrderByDescending(r => r.Id)
        .Select(r => r.ReservationNo)
        .FirstOrDefaultAsync();

    var seq = 1;
    if (last != null) seq = int.Parse(last[^4..]) + 1;

    // 2. 创建 Reservation 实体
    var reservation = new Reservation
    {
        UserId = userId,
        ReservationNo = $"R{DateTime.Now:yyyyMMdd}{seq:D4}",  // 如 R202606260001
        VisitorType = request.VisitorType,
        VisitorName = request.Name,
        VisitorPhone = request.Phone,
        VisitDate = request.VisitDate,
        TimeSlot = request.TimeSlot,
        Companions = request.Companions,
        StayDuration = request.StayDuration,
        Purpose = request.Purpose,
        Status = "pending",  // 初始状态：待审核
    };

    // 3. 保存到数据库（触发器自动写入 ReservationStatusLog）
    _db.Reservations.Add(reservation);
    await _db.SaveChangesAsync();
}
```

**数据库交互**：`INSERT INTO Reservations` → 触发器 `trg_Reservations_StatusChange` 自动插入 `ReservationStatusLog`（`FromStatus=NULL, ToStatus='pending'`）

#### 6.3.2 预约审核流程

```csharp
// ReservationService.ReviewAsync()
public async Task ReviewAsync(int reviewerId, int id, ReviewReservationRequest request)
{
    var reservation = await _db.Reservations.FindAsync(id)
        ?? throw new KeyNotFoundException("预约不存在");

    if (reservation.Status != "pending")
        throw new InvalidOperationException("只能审核待处理状态的预约");

    reservation.Status = request.Action;     // "approved" 或 "rejected"
    reservation.ReviewerId = reviewerId;
    reservation.ReviewRemark = request.Remark;
    reservation.ReviewedAt = DateTime.UtcNow;

    // 保存后触发器自动写入状态变更日志
    await _db.SaveChangesAsync();
}
```

**数据库交互**：`UPDATE Reservations SET Status='approved', ReviewerId=..., ...` → 触发器检测 Status 变更 → `INSERT INTO ReservationStatusLog(FromStatus='pending', ToStatus='approved', ...)`

#### 6.3.3 入校核验流程

```csharp
// EntryExitService.EntryAsync()
public async Task<EntryExitRecord> EntryAsync(int operatorId, EntryCheckRequest request)
{
    var reservation = await _db.Reservations.FindAsync(request.Id)
        ?? throw new KeyNotFoundException("预约不存在");

    // 业务规则：只有审核通过的预约才能入校
    if (reservation.Status != "approved")
        throw new InvalidOperationException("该预约未通过审核，不可入校");

    // 查找对应的校门
    var gate = await _db.Gates.FirstOrDefaultAsync(g => g.Name == request.Gate)
        ?? await _db.Gates.FirstAsync();

    // 更新预约状态
    reservation.Status = "checked_in";

    // 创建出入校记录
    var record = new EntryExitRecord
    {
        ReservationId = reservation.Id,
        UserId = reservation.UserId,
        EntryTime = DateTime.UtcNow,
        EntryGateId = gate.Id,
        OperatorId = operatorId,
    };

    _db.EntryExitRecords.Add(record);
    await _db.SaveChangesAsync();  // 一个事务完成两个操作
    return record;
}
```

**数据库交互**：`UPDATE Reservations SET Status='checked_in'` + `INSERT INTO EntryExitRecords(...)` 在同一事务中执行 → 触发器记录状态变更

#### 6.3.4 活动报名 — 并发控制

```csharp
public async Task RegisterAsync(int userId, RegisterActivityRequest request)
{
    // 1. 检查活动是否存在
    var activity = await _db.Activities.FindAsync(request.ActivityId)
        ?? throw new KeyNotFoundException("活动不存在");

    // 2. 检查名额（业务层控制，配合数据库 UNIQUE 约束兜底）
    if (activity.CurrentCount >= activity.MaxParticipants)
        throw new InvalidOperationException("报名人数已满");

    // 3. 检查是否重复报名
    if (await _db.ActivityRegistrations.AnyAsync(r =>
            r.ActivityId == request.ActivityId && r.UserId == userId))
        throw new InvalidOperationException("您已报名该活动");

    // 4. 创建报名记录 + 更新活动计数
    var registration = new ActivityRegistration { ... };
    activity.CurrentCount++;  // 计数 +1

    _db.ActivityRegistrations.Add(registration);
    await _db.SaveChangesAsync();  // 同一事务，保证计数准确性
}
```

**并发安全设计**：

- 业务层校验 `CurrentCount >= MaxParticipants`（乐观检查）
- 数据库层面 `UNIQUE(ActivityId, UserId)` 约束防止重复报名（兜底保障）
- EF Core 的 `SaveChangesAsync()` 在同一个数据库事务中完成"插入报名记录"和"更新活动计数"两个操作

#### 6.3.5 违规检测与黑名单自动化

系统设计了**三层违规检测机制**，通过 C# 代码（非 SQL 触发器）在关键页面访问时自动触发：

| 检测类型     | 触发条件                                        | 实现位置                                         | 说明                                                |
| ------------ | ----------------------------------------------- | ------------------------------------------------ | --------------------------------------------------- |
| 爽约检测     | 预约`status=approved` 且 `VisitDate < 今天` | `EntryExitService.AutoDetectViolationsAsync()` | 标记为`no_show`，插入 ViolationRecord             |
| 超时滞留检测 | 在校访客超过区域关闭时间                        | 同上                                             | 按`OpenRules.AreaId` 分组取最早关闭时间，避免重复 |
| 越权访问检测 | 访客类型无权限进入某区域                        | 同上                                             | 对比`AreaPermission.VisitorType`                  |

**违规→拉黑自动化链路**：

```
AutoDetectViolationsAsync()
  → ① 爽约检测 → UPDATE Reservations SET Status='no_show'
               → INSERT ViolationRecord (SourceType='system')
  → ② 超时检测 → INSERT ViolationRecord (SourceType='system', Location=区域名)
  → ③ 越权检测 → INSERT ViolationRecord (SourceType='system')
  → AutoBlacklistAsync()
    → SELECT UserId, COUNT(*) FROM ViolationRecords GROUP BY UserId HAVING COUNT(*)>=3
    → INSERT Blacklist (ExpiresAt = NOW + 3个月, IsActive=1)
```

**注意**：违规检测和拉黑逻辑均通过 **C# 代码实现**，而非 SQL 触发器。SQL 触发器仅用于 `ReservationStatusLog` 的状态变更审计日志记录。

### 7.2 路由设计与角色守卫

```javascript
// 访问控制：
// 1. 访客端 (/visitor/*) — 公开浏览，但"我的预约"和"个人中心"需登录
// 2. 管理员端 (/admin/*) — 必须登录 且 角色为 admin

// 路由守卫核心逻辑
router.beforeEach((to, from, next) => {
  if (to.meta.requiresAuth && !authStore.isLoggedIn) {
    next('/auth/login')           // 未登录 → 跳转登录页
  } else if (to.meta.role && authStore.user?.role !== to.meta.role) {
    next('/')                     // 角色不匹配 → 跳转首页
  } else {
    next()
  }
})
```

### 7.3 前后端通信

前端通过 Axios 实例统一管理 HTTP 请求，核心配置：

```javascript
// request.js — Axios 实例
const api = axios.create({
  baseURL: '/api',       // 通过 Vite 代理转发到后端 http://localhost:5000
  timeout: 10000,
})

// 请求拦截器：自动附加 JWT Token
api.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 响应拦截器：统一错误处理
api.interceptors.response.use(
  response => response.data,     // 直接返回 data
  error => {
    if (error.response?.status === 401) {
      // Token 过期 → 清除登录状态 → 跳转登录页
    }
    return Promise.reject(error)
  }
)
```

### 8.4 智能违规检测与黑名单机制

在完成前后端数据一致性修复后，我们进一步实现了**自动化违规检测**和**黑名单强制执行**两大增强功能。

#### 8.4.1 自动违规检测机制

系统通过 `EntryExitService.AutoDetectViolationsAsync()` 在管理员访问数据看板和安保查看在校访客时自动触发，涵盖三类检测：

```mermaid
flowchart TD
    A[触发入口] --> B{① 爽约检测}
    A --> C{② 按区域超时检测}
    A --> D{③ 访问权限检测}
  
    B -->|预约通过但未到校| B1[标记 status=no_show]
    B1 --> B2[创建 ViolationRecord]
  
    C -->|查询活跃 OpenRule| C1[按 AreaId 去重取关闭时间]
    C1 -->|当前时间 > 关闭时间| C2[查找在校访客]
    C2 --> C3[创建 overstay 违规<br/>含区域名+滞留时长]
  
    D -->|在校访客 VisitorType| D1[对比 AreaPermission]
    D1 -->|类型无权限进入| D2[创建 trespass 违规]
  
    B2 --> E[AutoBlacklistAsync]
    C3 --> E
    D2 --> E
    E -->|累计 ≥3 次| F[自动加入黑名单 3个月]
```

**实际检测结果**：

| 检测类型            | 检测对象                           | 结果      | 详情                                             |
| ------------------- | ---------------------------------- | --------- | ------------------------------------------------ |
| 爽约 (no_show)      | 张三、王五、孙七、安保张三等 5 人  | ✅ 已标记 | 过期未到校的 approved 预约                       |
| 超时滞留 (overstay) | 李四                               | ✅ 已检测 | 入校 7 天未离校，区域「全校」关闭，滞留 180 小时 |
| 权限违规 (trespass) | 李四                               | ✅ 已检测 | 访客类型无权限进入「封闭区域」                   |
| 自动拉黑            | 张三(3次)、赵六(3次)、李四(2→4次) | ✅ 已拉黑 | 3 个月有效期                                     |

#### 8.4.2 黑名单拦截机制

**登录拦截**（`AuthService.LoginAsync`）：

```
张三 (13800138001) 登录 → 查询 Blacklist → IsActive=1, ExpiresAt > now
→ 返回 401: "您已被列入黑名单，截止 2026-09-26 前不可使用本系统。原因：系统自动检测：累计违规3次"
```

**预约拦截**（`ReservationService.CreateAsync`）：

```
黑名单用户提交预约 → 检测到 Blacklist 记录 → 
→ 抛出异常: "您已被列入黑名单，XXXX-XX-XX 前无法预约"
```

#### 8.4.3 区域-规则-违规联动

此前区域管理（`CampusAreas`）、开放规则（`OpenRules`）和违规检测三个模块相互割裂。改造后形成联动：

| 改进项                     | 改动                              | 效果                                               |
| -------------------------- | --------------------------------- | -------------------------------------------------- |
| `OpenRule.AreaId`        | 新增 FK → CampusAreas            | 规则可绑定到具体区域，null=全校通用                |
| `CampusArea.*Start/*End` | 新增 4 个时段字段                 | 每个区域有独立默认开放时间                         |
| 违规检测去区域化           | 重写`AutoDetectViolationsAsync` | 按区域关闭时间检测超时，按 AreaPermission 检测越权 |
| 去重机制                   | GroupBy AreaId + Local 检查       | 同一区域多条规则不重复创建违规                     |

#### 8.4.4 举报→违规→拉黑完整链

```
用户提交 Report → 管理员审核通过 → ReportController.Review()
  → 自动创建 ViolationRecord（SourceType=report）
  → 统计该用户累计违规次数
    → ≥3 次 → 自动 Blacklist（Reason="累计违规X次（含：举报类型）"）

```

## 10. 前端-后端-数据库关联分析

本章选取系统中最具代表性的**三个关联业务行为**，详细分析从前端用户操作到后端业务逻辑再到数据库持久化的完整数据流。

### 10.1 业务链路一：访客预约 → 审核 → 入校 → 离校（全流程）

这是系统的**核心业务流程**，涉及 4 个角色（访客、管理员、安保 ×2 次）、4 个数据库表、1 个触发器和 5 次状态变更。

```mermaid
sequenceDiagram
    participant V as 访客前端
    participant API as 后端API
    participant DB as SQL Server

    Note over V,DB: === 第一步：提交预约 ===
    V->>API: POST /api/reservations<br/>{name, phone, visitorType, visitDate, timeSlot, ...}
    API->>DB: INSERT INTO Reservations<br/>Status='pending', ReservationNo='R202606260001'
    DB-->>API: 触发器: INSERT INTO ReservationStatusLog<br/>(FromStatus=NULL, ToStatus='pending')
    API-->>V: { message: "预约申请已提交" }

    Note over V,DB: === 第二步：管理员审核 ===
    V->>API: PUT /api/reservations/1/review<br/>{action: "approved", remark: "欢迎参观"}
    API->>API: 校验 Status=='pending'
    API->>DB: UPDATE Reservations<br/>SET Status='approved', ReviewerId=1, ReviewRemark='欢迎参观'
    DB-->>API: 触发器: INSERT INTO ReservationStatusLog<br/>(FromStatus='pending', ToStatus='approved')
    API-->>V: { message: "操作成功" }

    Note over V,DB: === 第三步：安保核验入校 ===
    V->>API: POST /api/entry-exit/search<br/>{query: "R202606260001"}
    API->>DB: SELECT * FROM Reservations<br/>WHERE ReservationNo='R202606260001'
    API-->>V: { 预约详情, status: "approved" }

    V->>API: POST /api/entry-exit/entry<br/>{id:1, gate:"南门"}
    API->>API: 校验 Status=='approved'
    API->>DB: UPDATE Reservations SET Status='checked_in'
    API->>DB: INSERT INTO EntryExitRecords<br/>(EntryTime=GETDATE(), EntryGateId=1, OperatorId=7)
    DB-->>API: 触发器: INSERT INTO ReservationStatusLog<br/>(FromStatus='approved', ToStatus='checked_in')
    API-->>V: { EntryExitRecord, message: "入校成功" }

    Note over V,DB: === 第四步：安保登记离校 ===
    V->>API: PUT /api/entry-exit/exit<br/>{name:"张三", gate:"南门"}
    API->>DB: SELECT * FROM EntryExitRecords<br/>WHERE Reservation.VisitorName='张三' AND ExitTime IS NULL
    API->>DB: UPDATE EntryExitRecords<br/>SET ExitTime=GETDATE(), ExitGateId=1
    API->>DB: UPDATE Reservations SET Status='checked_out'
    DB-->>API: 触发器: INSERT INTO ReservationStatusLog<br/>(FromStatus='checked_in', ToStatus='checked_out')
    API-->>V: { message: "离校登记成功" }
```

**数据库关联分析**：

| 步骤     | 涉及表                                              | 关联方式                                           | 数据一致性保障                                |
| -------- | --------------------------------------------------- | -------------------------------------------------- | --------------------------------------------- |
| 预约提交 | `Reservations` + `ReservationStatusLog`         | `ReservationId` 外键                             | 触发器自动记录，确保不遗漏                    |
| 审核     | `Reservations` + `Users`（审核人）              | `ReviewerId → Users.Id`                         | 外键约束 + 状态 CHECK 约束                    |
| 入校     | `Reservations` + `EntryExitRecords` + `Gates` | `ReservationId`, `EntryGateId`, `OperatorId` | 同一事务中完成状态更新和记录插入              |
| 离校     | `EntryExitRecords` + `Reservations`             | `ReservationId`                                  | 通过`ExitTime IS NULL` 条件确保只更新在记录 |

### 10.2 业务链路二：活动发布 → 报名 → 签到（并发控制）

```mermaid
sequenceDiagram
    participant A as 管理员前端
    participant V as 访客前端
    participant API as 后端API
    participant DB as SQL Server

    Note over A,DB: === 管理员发布活动 ===
    A->>API: POST /api/activities<br/>{title:"校园开放日", maxParticipants:100, ...}
    API->>DB: INSERT INTO Activities<br/>(Status='open', CurrentCount=0, MaxParticipants=100)
    API-->>A: { message: "活动已发布" }

    Note over V,DB: === 访客报名 ===
    V->>API: POST /api/activities/register<br/>{activityId: 1}
    API->>DB: SELECT * FROM Activities WHERE Id=1
    API->>API: 检查 CurrentCount(0) < MaxParticipants(100) ✓
    API->>API: 检查 UNIQUE(ActivityId, UserId) 无重复 ✓
    API->>DB: BEGIN TRANSACTION
    API->>DB:   INSERT INTO ActivityRegistrations (...)
    API->>DB:   UPDATE Activities SET CurrentCount = CurrentCount + 1
    API->>DB: COMMIT TRANSACTION
    API-->>V: { message: "报名成功" }

    Note over A,DB: === 管理员签到 ===
    A->>API: PUT /api/activities/checkin/1
    API->>DB: UPDATE ActivityRegistrations<br/>SET Status='checked_in', CheckedInAt=GETDATE()
    API-->>A: { message: "签到成功" }
```

**关键数据库设计**：

- `Activities.CurrentCount` 是**冗余字段**（可从 `ActivityRegistrations` 聚合计算），但保留它通过应用层同步更新，避免每次查询活动列表时都要 `COUNT` 子查询
- `UNIQUE(ActivityId, UserId)` 约束在数据库层面兜底防止重复报名
- 报名操作在**同一事务**中完成"插入报名记录"和"递增计数"，保证数据一致性

### 10.3 业务链路三：区域权限管控

```mermaid
sequenceDiagram
    participant A as 管理员前端
    participant API as 后端API
    participant DB as SQL Server

    Note over A,DB: === 配置区域权限 ===
    A->>API: POST /api/areas<br/>{name:"校史馆", type:"public", accessLevel:"restricted",<br/> allowedTypes:["alumni","study_group"]}
    API->>DB: INSERT INTO CampusAreas<br/>(Name='校史馆', AccessLevel='restricted')
    API->>DB: INSERT INTO AreaPermissions (AreaId=5, VisitorType='alumni')
    API->>DB: INSERT INTO AreaPermissions (AreaId=5, VisitorType='study_group')
    API-->>A: { message: "保存成功" }

    Note over A,DB: === 查询某区域允许的访客类型 ===
    API->>DB: SELECT * FROM CampusAreas<br/>LEFT JOIN AreaPermissions ON CampusAreas.Id = AreaPermissions.AreaId
    API-->>A: { 区域详情 + 权限列表 }
```

**多对多关系设计说明**：

- `CampusAreas` 和访客类型之间的多对多关系通过 `AreaPermissions` 中间表实现
- `UNIQUE(AreaId, VisitorType)` 约束确保同一区域对同一访客类型不会出现重复权限记录
- `ON DELETE CASCADE`：删除区域时自动清除相关联的权限记录，防止孤儿数据
- 这种设计使系统具备良好的扩展性——新增访客类型或区域时无需修改表结构，只需增删 `AreaPermissions` 记录

### 10.4 业务链路四：违规检测 → 自动拉黑 → 黑名单拦截

这是系统的**智能安全管控链路**，涉及 5 个数据库表，通过 C# 代码在页面访问时自动触发。

```mermaid
sequenceDiagram
    participant A as 管理员/安保前端
    participant API as 后端API
    participant SVC as EntryExitService
    participant DB as SQL Server

    Note over A,DB: === 触发检测（访问Dashboard或在校访客页面时自动执行） ===
    A->>API: GET /api/admin/dashboard 或 GET /api/entry-exit/current
    API->>SVC: AutoDetectViolationsAsync()

    Note over SVC,DB: === ①爽约检测 ===
    SVC->>DB: SELECT * FROM Reservations<br/>WHERE Status='approved' AND VisitDate < TODAY
    DB-->>SVC: 返回 3 条过期预约（张三/王五/孙七）
    SVC->>DB: UPDATE Reservations SET Status='no_show'
    SVC->>DB: INSERT INTO ViolationRecords<br/>(ViolationType='no_show', SourceType='system', SourceId=ReservationId)
    DB-->>SVC: 触发器: INSERT StatusLog (FromStatus='approved', ToStatus='no_show')

    Note over SVC,DB: === ②超时滞留检测（按区域关闭时间） ===
    SVC->>DB: SELECT * FROM OpenRules LEFT JOIN CampusAreas<br/>WHERE IsActive=1 AND StartDate<=TODAY AND EndDate>=TODAY
    DB-->>SVC: 返回 3 条活跃规则
    SVC->>SVC: 按 AreaId 去重取最早关闭时间
    SVC->>DB: SELECT * FROM EntryExitRecords<br/>WHERE EntryTime IS NOT NULL AND ExitTime IS NULL<br/>AND EntryTime < 关闭时间
    DB-->>SVC: 李四（滞留 180 小时）
    SVC->>DB: INSERT INTO ViolationRecords<br/>(ViolationType='overstay', Location='全校', SourceId=EntryExitId)

    Note over SVC,DB: === ③越权检测 ===
    SVC->>DB: SELECT * FROM AreaPermissions INNER JOIN CampusAreas
    DB-->>SVC: 返回所有区域权限配置
    SVC->>SVC: 对比在校访客 VisitorType 与禁止区域权限

    Note over SVC,DB: === 自动拉黑 ===
    SVC->>DB: SELECT UserId, COUNT(*) FROM ViolationRecords<br/>GROUP BY UserId HAVING COUNT(*)>=3
    DB-->>SVC: 张三:3次 ✓ / 赵六:3次 ✓ / 李四:3次 ✓
    SVC->>DB: INSERT INTO Blacklist<br/>(UserId, Reason='系统自动检测：累计违规3次',<br/>ExpiresAt=NOW+3个月, IsActive=1)
    DB-->>SVC: 3 人加入黑名单

    Note over A,DB: === 黑名单拦截效果 ===
    A->>API: POST /api/auth/login<br/>{phone:"13800138001", password:"123456"}
    API->>DB: SELECT * FROM Blacklist<br/>WHERE UserId=? AND IsActive=1 AND ExpiresAt>GETUTCDATE()
    DB-->>API: 张三在有效黑名单中
    API-->>A: 401 Unauthorized<br/>"您已被列入黑名单，截止 2026-09-26 前不可使用本系统"
```

**数据库关联分析**：

| 步骤     | 涉及表                                                   | 关联方式                              | 关键设计                                           |
| -------- | -------------------------------------------------------- | ------------------------------------- | -------------------------------------------------- |
| 爽约检测 | `Reservations` → `ViolationRecords`                 | `SourceId` → `Reservations.Id`   | 避免重复：检查是否已有同 UserId+Type+SourceId 违规 |
| 超时检测 | `OpenRules` ← `CampusAreas` → `EntryExitRecords` | `AreaId`, `EntryTime` vs 关闭时间 | 按 AreaId 去重，同一区域多个规则只取最早关门的     |
| 自动拉黑 | `ViolationRecords` → `Blacklist`                    | `UserId` GROUP BY                   | 拉黑前检查`Blacklist.IsActive` 避免重复拉黑      |
| 登录拦截 | `Blacklist` + `Users`                                | `UserId` JOIN                       | `ExpiresAt > GETUTCDATE()` 保证过期自动放行      |

### 10.5 业务链路五：举报 → 违规记录 → 黑名单

```mermaid
sequenceDiagram
    participant U as 用户前端
    participant API as 后端API
    participant DB as SQL Server

    Note over U,DB: === 提交举报 ===
    U->>API: POST /api/reports<br/>{targetName, violationType, location, description}
    API->>DB: INSERT INTO Reports<br/>(Status='pending')
    API-->>U: { message: "举报已提交" }

    Note over U,DB: === 管理员审核通过 ===
    U->>API: PUT /api/reports/1/review<br/>{status:"approved"}
    API->>DB: UPDATE Reports SET Status='approved', ReviewerId=?, ReviewedAt=NOW
    API->>DB: SELECT * FROM Users WHERE Name=目标人姓名
    API->>DB: INSERT INTO ViolationRecords<br/>(UserId=目标ID, ViolationType=同举报类型,<br/>SourceType='report', SourceId=ReportId)
    API->>DB: SELECT COUNT(*) FROM ViolationRecords WHERE UserId=?
    DB-->>API: 累计 3 次
    API->>DB: INSERT INTO Blacklist<br/>(UserId, Reason='累计违规3次（含：XXX）')
    API-->>U: { message: "审核完成，已创建违规记录并拉黑" }
```

**数据库关联分析**：

| 步骤       | 涉及表                                   | 关联方式                                       |
| ---------- | ---------------------------------------- | ---------------------------------------------- |
| 举报→违规 | `Reports` → `ViolationRecords`      | `SourceType='report'`, `SourceId=ReportId` |
| 违规→拉黑 | `ViolationRecords` → `Blacklist`    | `GROUP BY UserId HAVING COUNT(*)>=3`         |
| 同名匹配   | `Reports.TargetName` → `Users.Name` | `FirstOrDefault` 查找目标用户                |

### 10.6 业务链路六：开放规则 × 区域联动 → 违规检测

```mermaid
sequenceDiagram
    participant A as 管理员前端
    participant API as 后端API
    participant DB as SQL Server

    Note over A,DB: === 创建区域专属规则 ===
    A->>API: POST /api/open-rules<br/>{areaId:5, dateType:"exam", timeSlot:"afternoon",<br/>afternoonStart:"14:00", afternoonEnd:"16:00"}
```

**数据库设计要点**：

- `OpenRules.AreaId`（NULL=全校通用，FK→CampusAreas ON DELETE SET NULL）
- 同一区域多条规则时按 `AreaId` 分组取最早 `AfternoonEnd`，避免重复创建违规
- 违规记录的 `Location` 字段存储区域名称（如"校史馆"），便于追溯


### 11.2 技术亮点

1. **触发器实现无侵入审计**：预约状态变更日志完全由数据库触发器自动记录，应用层无需关心，任何途径修改状态都能被追溯
2. **CHECK 约束的枚举值控制**：所有状态字段都在数据库层面通过 CHECK 约束限制合法取值，比纯应用层校验更可靠
3. **多对多权限模型**：`AreaPermissions` 中间表实现区域与访客类型的灵活权限配置，`OpenRules.AreaId` 实现区域级差异化时段管理
4. **前后端分离 + JWT 认证**：符合现代 Web 开发最佳实践，前后端独立部署、独立开发
5. **分层架构**：Controller → Service → EF Core → Database，职责清晰，便于测试和维护
6. **智能违规检测**：C# 代码实现每日自动检测（非 SQL 触发器），涵盖爽约、超时滞留、越权访问三类违规，按区域关闭时间精准判断
7. **自动化黑名单**：违规→记录→累计≥3次→自动拉黑→登录拦截→预约拒绝，全程无需人工干预

### 11.3 不足与改进方向

| 不足                                   | 改进方向                                                      |
| -------------------------------------- | ------------------------------------------------------------- |
| 活动报名在高并发下可能出现超卖         | 使用数据库行锁（`SELECT ... FOR UPDATE`）或乐观锁（版本号） |
| 预约编号由应用层生成，存在并发冲突风险 | 改用数据库 SEQUENCE 或`GenerateReservationNo` 函数          |
| 缺少数据可视化图表                     | 集成 ECharts 实现客流趋势图和访客类型饼图                     |
| 缺少文件上传功能                       | 活动封面图、举报证据等目前仅支持 URL，可扩展为文件上传        |
| 缺少消息通知                           | 预约审核结果可通过短信或站内信通知访客                        |
